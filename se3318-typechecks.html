Software Construction: Coding Standards & Checkstyle Reference Sheet

Comprehensive guide to code quality rules and best practices

This reference sheet compiles key Checkstyle rules and programming style guidelines from your course material. Understanding these rules, their purpose, common violations, and correct usage is crucial for writing high-quality, maintainable software and for your final exam.

1. Class-Level Checks

These checks apply to the overall structure, naming, and dependencies of classes.

Rule Name Where It Is Used What It Checks Why It Matters Common Violations Correct Usage Example
VisibilityModifier Class (Fields) Ensures proper visibility of class members. By default, fields should be `private`. Only `static final` or truly immutable fields may be `public`. Enforces good encapsulation; hides implementation details; reduces external dependencies. 
public int num1; // non-final public field
protected String field2; // protected field often disallowed
 
class Example {
  private int myPrivateField;
  public static final int CONSTANT = 10; // OK
  // public final ImmutableList<String> items; // OK if ImmutableList is configured as immutable
}
FinalClass Class Verifies that classes not intended for inheritance are declared `final`, especially those with only private constructors. Prevents unintended or incorrect subclassing; clarifies design intent. 
class Test { private Test() {} } // Violation: Should be final
 
final class Test { private Test() {} } // OK
class MyClass { public MyClass() {} } // OK: Designed for instantiation/extension
HideUtilityClassConstructor Class Ensures that utility classes (containing only static members) have a private constructor to prevent instantiation. For extensible utilities, use `protected` ctor throwing `UnsupportedOperationException`. Prevents meaningless object instantiation; clarifies class role. 
class Utils { public Utils() {} public static void doSomething() {} } // Violation
 
class Utils { private Utils() {} public static void doSomething() {} } // OK
class ExtensibleUtils { protected ExtensibleUtils() { throw new UnsupportedOperationException(); } public static void anotherMethod() {} } // OK (for extension)
DesignForExtension Class (Methods) Checks that overridable methods with non-empty implementations have Javadoc. Enforces providing empty "hook" methods for subclasses. Protects superclasses from being broken by subclasses; ensures clear API documentation. 
class Base { public void doSomething() { /* ... */ } } // Violation: Overridable without Javadoc
 
class Base { /** Method documentation. */ public void doSomething() { /* ... */ }
  protected void hookMethod() {} // OK: Empty hook
}
AbstractClassName Class Ensures abstract class names conform to a specified pattern (e.g., starting with "Abstract") and are declared `abstract`. Improves naming consistency; clarifies class hierarchy. 
abstract class MyBaseClass {} // Violation (if pattern requires "Abstract")
class AbstractService {} // Violation (if not abstract)
 
abstract class AbstractProcessor {} // OK
ClassDataAbstractionCoupling (CDAC) Class Measures the number of unique other classes instantiated or referenced as members/local variables within a class. Reduces coupling; high DAC increases maintenance costs. A class that imports and uses instances of 10+ different custom types. Strive for a low DAC score (e.g., max 3-5 unique types). Refactor large classes into smaller, more focused ones.
InterfaceIsType Class (Interface) Enforces that interfaces define methods and primarily serve as types, not just constant holders. Promotes proper interface usage (defining behavior, not just data). 
interface Constants { int MAX = 10; String NAME = "App"; } // Violation
 
interface Service { void perform(); int calculate(); } // OK
interface Marker {} // OK (if marker interfaces allowed)
ClassFanOutComplexity (CFO) Class Measures the number of other types (classes, records, interfaces, enums, annotations) a given class directly relies on. Reduces coupling; high CFO significantly increases maintenance burden. A class with numerous imports and direct dependencies on many different types. Strive for low CFO. Break down large, complex classes into smaller, more manageable components with fewer dependencies.
AvoidStarImport Class (Imports) Prohibits the use of `import *` (wildcard imports) for packages or static members. Reduces coupling; prevents naming conflicts; improves clarity of class origin. 
import java.util.*;
import static java.lang.Math.*;
 
import java.util.ArrayList;
import java.util.HashMap;
import static java.lang.Math.PI;
MissingCtor Class Ensures that non-abstract classes explicitly define a constructor, rather than relying solely on the implicit default no-argument constructor. Makes instantiation clearer; allows for specific initialization logic. 
class MyClass { int value; } // Violation: No explicit constructor
 
class MyClass { int value; public MyClass() { this.value = 0; } } // OK
MethodCountCheck Class Limits the number of methods declared in each type, by access modifier or total count. (Methods in nested enums/anonymous classes are not counted). Helps prevent "God Objects" or classes with too many responsibilities; enforces Single Responsibility Principle. A class with excessively many methods (e.g., 50+ public methods). Classes should be focused on a single responsibility, leading to a manageable number of methods. Refactor if method count is too high.
EmptyLineSeparator Class/File Enforces empty lines between different code blocks (e.g., package, imports, fields, constructors, methods, nested classes, initializers). Improves code readability and visual organization. No empty lines separating imports from fields, or fields from constructors. 
package com.example;

import java.util.List;

public class MyClass {
  private int count; // Fields

  public MyClass() { // Constructor
    this.count = 0;
  }

  public void doSomething() { // Method
    // ...
  }
}
JavaNCSS Class/File Counts non-commenting source statements (NCSS) for methods, classes, and files. Measures code complexity; high NCSS indicates a need for refactoring. Very large files or classes with thousands of lines of code. Break down large files and classes into smaller, more manageable units. Keep methods concise.

2. Method-Level Checks

These checks focus on the design, naming, parameters, and internal structure of individual methods.

Rule Name Where It Is Used What It Checks Why It Matters Common Violations Correct Usage Example
MissingOverride Method Verifies that the `@Override` annotation is present when the `@inheritDoc` Javadoc tag is used, or when a method intends to override a superclass method. Clearly indicates overriding intent; helps catch accidental overloading instead of overriding. 
class Sub extends Super { /** @inheritDoc */ public void doIt() {} } // Violation
 
class Sub extends Super { @Override /** @inheritDoc */ public void doIt() {} } // OK
OverloadedMethodsDeclarationOrder Method Checks that overloaded methods (same name, different parameters) are grouped together in the class declaration. Improves code readability and maintainability; easier to find related methods. 
public void foo(int i) {} public void bar() {} public void foo(String s) {} // Violation
 
public void foo(int i) {} public void foo(String s) {} public void bar() {} // OK
MethodName Method Enforces naming conventions for methods (e.g., camel case, starting with a lowercase letter, strong verb followed by object, describes all side effects and return values). Improves readability, consistency, and clarity of intent. 
public void DoSomething() {} // Violation
public void handle() {} // Violation: Meaningless verb
 
public void calculateTotalAmount() {} // OK
public boolean isValidated() {} // OK
public Document printDocument() {} // OK
MethodLength Method Limits the maximum number of lines in a method. (Can be configured to count comments/empty lines or only code lines). Promotes shorter, more focused routines; improves readability, testability, and reduces complexity. A method spanning hundreds of lines. Extract long methods into smaller, functionally cohesive routines. Aim for methods that fit on a single screen.
MethodParamPad Method (Parameters) Checks the spacing between the method identifier and the opening parenthesis `(`. Enforces consistent code formatting and readability. 
public void myMethod () {} // Violation
public void anotherMethod( ) {} // Violation
 
public void myMethod() {} // OK
public void anotherMethod(int param) {} // OK
MethodTypeParameterName Method (Generics) Checks that method type parameter names (for generics) conform to a specified pattern (usually single uppercase letters, e.g., `T`, `K`, `V`). Ensures consistency and readability for generic code. 
public <type> void process(type item) {} // Violation
 
public <T> void process(T item) {} // OK
public <K, V> Map<K, V> createMap() {} // OK
FinalParameters Method (Parameters) Requires method parameters to be declared as `final`. Prevents accidental modification of parameters; makes code clearer (parameter values won't change). 
public void update(int value) { value++; } // Violation (if value not final)
 
public void update(final int value) { /* value cannot be modified */ } // OK
ParameterName Method (Parameters) Enforces naming conventions for method parameters (e.g., camel case, starting with lowercase). Improves readability and consistency. 
public void process(int MyParam) {} // Violation
 
public void process(int myParam) {} // OK
ParameterNumber Method (Parameters) Limits the number of parameters a method or constructor can have. Encourages simpler interfaces; reduces method complexity; indicates potential need for refactoring. A method with 8 or more parameters. Refactor methods with too many parameters by grouping related parameters into a new object (Parameter Object pattern).
RequireThis Method Checks that references to instance variables and methods of the current object explicitly use `this.`. Avoids ambiguity when local variables or parameters have the same name as instance members. 
class MyClass { int count; void increment() { count++; } } // Violation (if requireThis is true)
 
class MyClass { int count; void increment() { this.count++; } } // OK
ReturnCount Method Restricts the number of `return` statements in methods, constructors, and lambda expressions. Promotes clearer control flow; reduces complexity by having fewer exit points. A method with many `return` statements scattered throughout its logic. Limit returns to one or a few clearly defined exit points. Use `if/else` logic to consolidate return values.

3. Variable-Level Checks

These checks focus on the declaration, naming, initialization, and usage of variables.

Rule Name Where It Is Used What It Checks Why It Matters Common Violations Correct Usage Example
MemberName Variable (Instance) Checks that instance variable names conform to a specified pattern (e.g., camel case, starting with lowercase, not all uppercase). Improves readability and consistency across the codebase. 
public int NUM1; // Violation
private int Bad_Name; // Violation
 
public int totalCount; // OK
protected String customerName; // OK
LocalVariableName Variable (Local) Checks that local, non-final variable names conform to a specified pattern (e.g., camel case, starting with lowercase). Catch parameters are considered local variables. Improves readability and consistency for local scope. 
void myMethod() { int VAR = 10; } // Violation
 
void myMethod() { int counter = 0; } // OK
StaticVariableName Variable (Static) Checks that static, non-final variable names conform to a specified pattern (e.g., camel case, starting with lowercase). Improves readability and consistency for static members. 
public static int ItStatic1 = 2; // Violation
 
public static int currentUsers = 0; // OK
LocalFinalVariableName Variable (Local Final) Checks that local `final` variable names conform to a specified pattern (e.g., camel case, starting with lowercase). Catch parameters and resources in `try` statements are considered local final variables. Improves readability and consistency. 
final int VAR1 = 5; // Violation
 
final int maxAttempts = 3; // OK
FinalLocalVariable Variable (Local) Checks that local variables whose values are never changed are declared `final`. (Can also check unchanged parameters). Enforces immutability; prevents accidental modification; improves code clarity. 
void foo() { int result = 0; result = 1 + 2; } // Violation (if result is never changed after first assignment)
 
void foo() { final int result = 1 + 2; } // OK
MultipleVariableDeclarations Variable (Declaration) Enforces that each variable declaration is in its own statement and on its own line. Improves readability and clarity of declarations. 
int lower, higher; // Violation
int value, index; // Violation
 
int lower;
int higher;
ExplicitInitialization Variable (Initialization) Checks if variables are explicitly initialized to their default values (e.g., `0` for int, `null` for objects, `false` for boolean). Avoids redundant code, as Java automatically initializes variables to defaults. 
private int intField1 = 0; // Violation
private Obj objField1 = null; // Violation
 
private int intField3; // OK (relies on default init)
private int intField2 = 1; // OK (non-default init)
VariableDeclarationUsageDistance Variable (Usage) Checks the distance (number of statements) between a variable's declaration and its first usage. Reduces variable "live time"; improves readability by keeping related code together. 
int num; // declared
System.out.println("S1");
System.out.println("S2");
num = 1; // first use (large distance)
 
int num = 1; // declared and used immediately
UnusedLocalVariable Variable (Local) Checks that a local variable is declared and/or assigned, but never used. Promotes cleaner code; avoids dead code and potential errors from unused variables. 
void method() { int temp = 10; } // Violation: temp is unused
 Remove unused variables. If the variable is truly needed later, ensure it's referenced.

4. Loop/Control-Flow-Specific Checks

These checks ensure proper and predictable behavior of loops and conditional structures.

Rule Name Where It Is Used What It Checks Why It Matters Common Violations Correct Usage Example
ModifiedControlVariable Loop (`for`) Checks that the control variable of a `for` loop is not modified inside the loop block. Prevents unpredictable loop behavior and hard-to-debug logic errors. 
for (int i = 0; i < 10; i++) { i++; } // Violation
 
for (int i = 0; i < 10; i++) { /* Use i, but don't change it */ } // OK
MissingSwitchDefault Control Flow (`switch`) Checks that a `switch` statement has a `default` clause. Ensures all cases are handled (even unexpected ones); improves robustness and documentation. 
switch (value) { case 1: break; } // Violation
 
switch (value) { case 1: break; default: handleUnknown(); break; } // OK
FallsThrough Control Flow (`switch`) Checks for fall-through in `switch` statements where a `case` block contains code but lacks an explicit exit (e.g., `break`, `return`, `throw`). Prevents unintentional bugs due to fall-through; enforces clear control flow. 
switch (i) { case 1: i++; case 2: i++; break; } // Violation
 
switch (i) { case 1: i++; // falls through, necessary to increase i by one. case 2: i++; break; } // OK (with comment)

5. Java-Specific Checks

These checks enforce Java-specific idioms and best practices related to core language features.

Rule Name Where It Is Used What It Checks Why It Matters Common Violations Correct Usage Example
CovariantEquals Method Checks that if a class defines an `equals()` method with a covariant parameter type (e.g., `equals(MyClass other)`), it *also* overrides the standard `equals(Object obj)`. Prevents subtle bugs due to method overloading instead of overriding when `equals()` is used polymorphically. 
class Test { public boolean equals(Test i) { return false; } } // Violation
 
class Test { public boolean equals(Test i) { return false; } public boolean equals(Object i) { return false; } } // OK
SuperClone Method Ensures that an overriding `clone()` method invokes `super.clone()`. Essential for proper object cloning in an inheritance hierarchy to correctly copy superclass state. 
class Sub extends Super { public Object clone() { return new Sub(); } } // Violation
 
class Sub extends Super { @Override public Object clone() throws CloneNotSupportedException { return super.clone(); } } // OK
SuperFinalize Method Checks that an overriding `finalize()` method invokes `super.finalize()`. (Note: `finalize()` is generally discouraged in modern Java; prefer `try-with-resources` or explicit cleanup). Ensures proper resource cleanup in the inheritance chain. 
class B { protected void finalize() { System.out.println("cleanup"); } } // Violation
 
class A { @Override protected void finalize() throws Throwable { System.out.println("In finalize block"); super.finalize(); } } // OK
